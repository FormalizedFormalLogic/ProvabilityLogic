# SeqPL プロジェクト指示

## Lean の証明作業のワークフロー

sorry を埋める，補題を証明する，リファクタするなど Lean の証明作業を行うときは，以下に従う．

- **必ず worktree を切って作業する**．`main` で直接作業しない．ブランチを切るだけで worktree を作らない，というのも不可．`gwq add -b <branch>` で作業用ブランチと worktree を作成してから始める．
- **worktree 作成後，必ずプロジェクトルートの `.lake` を worktree へコピーする**（`cp -al` でハードリンク複製する）．コピーしないと Mathlib と FFL のビルドに凄まじい時間がかかってしまう．
- **証明には `/lean4:autoprove` スキルを使う**．sorry を埋める・補題を証明する際は autoprove で進める．
- **タスクの種類に応じてサブエージェントのモデルを使い分ける**．
  - **より難易度の高い形式証明（新規の sorry 埋め・補題証明など）には Fable を使う**．サブエージェントを起動する際も `model: fable` を指定する．
  - **形式証明のリファクタリング（既存証明の整理・補題の分割・命名の整理など）には Opus を使う**．Sonnet ではリファクタリングの質が不十分だったため，`model: opus` を指定する．
  - **それ以外の日常的なタスク**（`Foundation` など他のコードを参考として調べる，論文 PDF をスキャンするといった軽い調査作業）**には低コストのモデル（Sonnet・Haiku）を使う**．
- **Fable をモデルとするサブエージェントを起動する際は，auto-edit（自動承認）モードであっても必ず事前にユーザーの承認を取る**．コストの高いモデルであるため，起動前に対象タスク・見積もりを提示し，ユーザーが明示的に許可してから起動すること．
- **一つの補題（sorry）を埋める度にコミットする**．まとめて一度にコミットしない．
- **全ての作業が完了しても，勝手に `main` へマージしない**．ブランチ名・変更内容・ビルド確認結果を報告し，ユーザーの確認を待つ．承認されたら `main` に squash マージし，マージ後は作業ブランチを削除する．
- コミット・マージの前に該当モジュールが `lake build` で通ること，および LSP 診断にエラー・警告（残存 sorry を含む）が無いことを確認する．
- **worktree での作業を終える際は，必ず `lake exe mk_all --module` を実行して `SeqPL.lean` を更新する**．新しいファイルを追加した場合の import 漏れを防ぐため．これは今後ユーザーの許可なく実行してよい．
- **worktree に入る際は `EnterWorktree` ツールを `path` 引数付きで呼び，セッションの作業ディレクトリ自体を切り替える**．`gwq add -b <branch>` でブランチと worktree を作成し `.lake` をハードリンク複製した後，`gwq get <branch>` で得た worktree の絶対パスを `EnterWorktree` の `path` に渡す．これにより以降のツール呼び出しで毎回 `cd` を書く必要がなくなる．`EnterWorktree` の `name` 引数（新規 worktree 自動作成）は `.lake` 複製手順を経ないため使わない．作業終了時は `ExitWorktree` を `action: "keep"` で呼んで元のディレクトリへ戻る．
- **worktree や main のパスが必要な場面では，常に `gwq get <pattern>` で取得する**．`~/ghq/...` のような絶対パスを手で組み立てて使わない（`~` 展開の不整合や誤ったパス指定を防ぐため）．`.lake` の複製先・`.direct` のシンボリックリンク先・`EnterWorktree` の `path` など，パスを扱う全ての場面に適用する．

### worktree での作業と `.direct/` による進捗共有

`.direct/` は git 管理対象外（`.gitignore` 済み）のディレクトリであり，全 worktree・元のディレクトリ間でファイルシステムレベルに共有することで，異なるセッション・エージェント間の進捗共有に使う．

- **ユーザーから指示を受けたら，必ず新しい worktree を作って作業を始める**（Lean の証明作業に限らず，リファクタ・ドキュメント修正など全ての作業に適用する）．タスク内容を表す適当な slug（kebab-case）を決めてブランチ・worktree を作成する．
- **worktree 作成後（`.lake` のハードリンク複製と同じタイミングで），必ず `.direct` をシンボリックリンクに置き換える**．worktree 内に checkout された `.direct`（存在する場合）を削除し，元のリポジトリルート（`main` ブランチの worktree）の `.direct` へのシンボリックリンクを貼る．
  ```
  rm -rf $(gwq get <branch>)/.direct
  ln -s $(gwq get main)/.direct $(gwq get <branch>)/.direct
  ```
  これにより，どの worktree から `.direct/` を読み書きしても実体は同一ディレクトリを指すため，コミット・マージを待たずに他の worktree やエージェントの状況をリアルタイムに把握できる．
- **worktree を作成したら，実際の作業（証明・編集など）に着手する前に，必ず最初に以下の2つを行う**．
  1. **`.direct/worktrees.md` にチェックボックスのリストアイテムを追加する**．slug と何をするかを端的に書く（例: `- [ ] my-slug — ○○の証明`）．作業が完了したらチェックを入れる．
  2. **`.direct/{{slug}}.md` を作成し，このworktreeで何をするか（タスク内容・方針・現状）を書き込む**．
- **`.direct/{{slug}}.md` は基本的に 1 コミットごとに更新し，他のセッションやエージェントが `.direct/worktrees.md` を見れば状況を把握して共同作業できる状態を保つ**．シンボリックリンクにより即座に他の worktree からも見えるため，マージ完了を待つ必要はない．

## 論理式の変数命名

- 様相論理について：
  - 論理式には `A`, `B`, `C` などの最初のほうの Latin アルファベットを使う．
  - 論理式のリストまたは有限集合には `Γ`, `Δ` を使う．
  - 論理式の集合には `X`, `Y`, `Z` を使う．
- 1 階述語論理の論理式には `φ`, `ψ` などのギリシャ文字を使う．

## 言語について

- コード中のコメント（Lean のドキュメントコメント `/-- -/` やインラインコメントなど）は英語で書く．
- ユーザーとの応答（チャット・報告など）は引き続き日本語で書く．

## 参考実装

- 証明の方針に迷ったら，依存先の `Foundation`（`.lake/packages/Foundation/`）の対応する補題の証明を参考にする．
- `Foundation` の Vorspiel にある補題で SeqPL に無いものは，必要に応じて `SeqPL/Vorspiel/` に移植してよい．
